# Handoff — 2026-07-13 session (location inputs + GPS serial)

Written for picking this up in a fresh Claude Code session (different machine,
no conversation memory). Read this first, then check `git log` for the actual
diffs — this is a summary, not a replacement for reading the code.

## What this session built

PLAN.md §5 ("Location inputs") was previously just placeholders for two of the
four planned observer-location sources. This session wired up the remaining
two, plus a related fix and a new clock feature:

1. **Browser geolocation** (`navigator.geolocation`) — `TopBar.svelte`'s
   `useBrowserLocation()`. Secure-context check, loading/error states.
2. **USB/serial NMEA GPS** (Web Serial) — new `app/src/serial/` module:
   - `nmea.ts` — pure GGA/RMC parser, checksum-validated, talker-ID-agnostic
     (GP/GN/GL/...). Unit-tested (`nmea.test.ts`) against published examples.
   - `nmeaFix.ts` — pure reducer merging GGA+RMC into one running fix.
     Freezes last-known lat/lon/altitude on fix loss rather than nulling it
     (mirrors real marine/aviation GPS UX). Unit-tested (`nmeaFix.test.ts`).
   - `connection.ts` — the real `navigator.serial` glue (untestable, kept
     thin). Connect/disconnect, baud selection (4800–115200), a manual
     Freeze/Unfreeze toggle (locks the observer marker against a jittery
     fix), a throttled store-flush (see **Buffer overrun**, below), and GPS
     clock discipline (below).
   - `web-serial.d.ts` — hand-written ambient TS types (Web Serial isn't in
     TS's DOM lib).
3. **GPS-disciplined clock** (PLAN.md §6) — `stores/clock.ts`'s
   `effectiveTime` applies a GPS-vs-system offset while connected (shown as
   a "GPS Δ" badge in `TopBar.svelte`), computed from RMC/GGA time fields.
   The offset is a plain module variable in `connection.ts`
   (`getGpsClockOffsetMs()`), not a store field, specifically so a fast
   receiver disciplining it doesn't add extra recomputation beyond
   `effectiveTime`'s existing 1Hz tick.
4. **Elevation out-of-bounds handling** — `data/elevation.ts`'s DEM grid is
   clipped to Spain/W. Mediterranean. `stores/observer.ts`'s `setObserver()`
   now flags (`elevationOutOfBounds`) and defaults to 0 instead of an
   edge-clamped guess when a location falls outside that grid, and the
   elevation box becomes editable regardless of source. A real GPS altitude
   reading (`gpsAltitudeM` param) bypasses this entirely — it's never a
   guess. The warning clears once the user supplies their own figure
   (`elevationManuallySet`) without re-locking the box.

## Buffer overrun — the bug, and where the fix actually stands

Reported on real hardware (a USB GPS receiver, 115200 baud) on first connect.

**First attempt** (insufficient): Chrome's default Web Serial receive buffer
is 255 bytes; bumped to 4096 via `bufferSize` in the `open()` call. Still
overran on real hardware.

**Root cause** (commit `d13de61`): `flushFix()` called `setObserver()`
synchronously, which cascades into Svelte's `derived` stores —
`localCircumstances.ts` (iterative Besselian root-finding +
astronomy-engine's `SearchRiseSet`) and `skyView.ts` (per-star/planet
position for the whole catalog) — genuinely expensive synchronous work (tens
to hundreds of ms). The throttle had a bypass: any `hasFix` transition
flushed *immediately*, no rate limit. Indoors during acquisition a marginal
signal flaps between fix/no-fix rapidly, so that bypass could fire the
expensive cascade many times back-to-back, blocking the async read loop long
enough that the browser's own serial receive buffer overran regardless of
size (backpressure propagates all the way back to the hardware buffer once
JS stops draining `reader.read()` promptly).

**Fix applied**:
- Throttle is now uniform — no more transition bypass. Losing/gaining lock
  can lag up to `FLUSH_INTERVAL_MS` (500ms) in the status line now; accepted
  as the trade for the connection staying up.
- `setObserver()` itself is deferred via `setTimeout(fn, 0)` in `flushFix()`
  — decouples it from the tight read-loop call stack instead of nesting
  inside it.
- `bufferSize` bumped further to 16384 as defense-in-depth on top of both.

**Verified**: full test suite (80/80), `svelte-check`/`tsc` (0 errors), and
live in-browser with a *mocked* `SerialPort` — fed a burst of 30 rapidly
alternating fix/no-fix sentences (the exact failure pattern) and confirmed
exactly one applied update resulted (the first line's value), not one per
transition; confirmed normal updates still flow after the throttle window
clears.

**NOT YET VERIFIED: real hardware.** Everything above was tested via a
JS-level mock of `navigator.serial` (`requestPort`/`open`/`readable` all
faked), since no real GPS device was available in the environment that did
this work. **This is the first thing to test when picking this up.**

### If it still overruns on real hardware

Things to try, roughly in order of how likely/cheap they are:
1. Bump `FLUSH_INTERVAL_MS` higher (e.g. 1000ms) — cheap, still correct.
2. Bump `SERIAL_BUFFER_SIZE` higher still (e.g. 65536) — cheap, no downside
   besides a larger in-memory buffer.
3. Check whether `localCircumstances`/`skyView`'s recompute is actually the
   bottleneck by timing it directly (`performance.now()` around the
   `setObserver` call in a throwaway build) rather than assuming — the
   diagnosis above is reasoned from code inspection, not measured on a real
   device.
4. If the receiver is configurable, reducing its output rate (1Hz instead of
   10Hz) or disabling unused constellations/sentence types (GSV in
   particular is verbose) would reduce data volume directly at the source.
5. Consider moving the astronomy recompute off the main thread (a Web
   Worker) if it's confirmed to be the real bottleneck — bigger change, only
   worth it if 1–4 don't fully resolve it.

## Environment notes (if starting fresh on another machine)

- If `npm`/`node` aren't on Windows PATH, check WSL — this session found
  Node v24 already installed there via `nvm`, unrelated to anything on
  Windows.
- Running `vite` (dev or build) directly against a **OneDrive-synced** path
  from WSL can hit `EIO` errors (Node's ESM loader doing sync reads over the
  9p/drvfs bridge). Workaround used this session: mirror `app/` into a
  WSL-native path (e.g. `~/work/eclipse-app`, excluding `node_modules`/
  `dist`), `npm ci` + `npm run dev`/`build` there, then copy `dist/` back to
  the real OneDrive path afterward if needed (e.g. for deploy).
- Web Serial requires a Chromium browser (Chrome/Edge) and either HTTPS or
  localhost.
