## 1. Executive summary

The engineering quality of what exists is **high** — better than the project's own
docs claim. The Besselian math core is oracle-validated to sub-millisecond contact
times, real bugs (the NaN terminator flicker, the ΔT skew, the after-sunset contacts)
were root-caused and locked in with regression tests, the shipped bundle makes
genuinely **zero runtime network requests**, and all data files are bundled, licensed
(with one exception), and generated by committed tooling.

The problem is **allocation, not execution**. Three days of intense effort went
overwhelmingly into map/sky rendering polish, while every feature that only matters
*on eclipse day in a field in Spain* remains unbuilt, and the deployment path the
plan itself calls a **deal-breaker** remains untested. With 30 days left, the review's
core recommendation is to invert the priority order: field-critical I/O, alerting,
packaging, and rehearsal first; rendering refinement last or never.

## 3. Field & deployment readiness

The runtime bundle is genuinely offline-clean (§4 confirms), but "the app makes no
network requests" is not the same as "the event-day deployment works." Every gap
here is verified, not speculative:

1. **No service worker / PWA manifest** — confirmed absent from `app/src`,
   `app/public`, and the built `dist/`. Consequence: an offline page reload works
   only while the local HTTP server is running. That's acceptable *if* the
   localhost-server path is the committed plan — but then the server must be part
   of the rehearsed, scripted setup. Alternative: `vite-plugin-pwa` is ~an hour of
   work and makes the app reload-proof once cached.
2. **`file://` is broken**, contradicting PLAN §2's "quick peek" row (PLAN.md:80).
   `dist/index.html:8–9` emits absolute `/assets/…` URLs (verified). Either set
   `base: './'` in `vite.config.ts` (one line — worth doing regardless, as a
   last-ditch fallback that makes the dist folder double-clickable) or delete the
   claim from the plan.
3. **No committed field-run script.** Add e.g. `app/serve-field.cmd`:
   `python -m http.server 8000 --bind 127.0.0.1 --directory dist` plus a
   `start http://localhost:8000`. One file, removes one day-of memory dependency.
4. **The repo — including `node_modules` and `dist` — lives inside OneDrive.**
   Files-on-demand can dehydrate files to placeholders; an offline field laptop
   cannot rehydrate them. OneDrive also rewrites mtimes (breaking incremental
   builds) and can fork files on sync conflict. **Before travel: copy the built
   `dist/` + server script to a non-synced path (e.g. `C:\eclipse-field\`), mark
   the repo "Always keep on this device," and rehearse from the copy.**
5. **A second, orphaned full repo copy** sits at
   `.claude/worktrees/agent-a861cfdb1a408d777/` — with its own `node_modules` —
   no longer registered with `git worktree list`, but still syncing through
   OneDrive. Delete it (§10).
6. **The Tauri spike** (again — §2.1). The plan's own words: run it early or lose
   the pivot window. T−30 is the last comfortable moment.
7. **No CI** — no `.github/` exists. With one developer and a hard deadline this is
   tolerable, but a 10-line workflow running `npm run check && npm run test` on push
   would catch a broken lockfile/regression while there's still time to notice.
   Optional; the rehearsals matter more.

## 4. Licensing & attribution — one hard defect

`app/src/data/NOTICE.md` covers 10 of the 11 shipped data files with unusual care.
The exception, verified by grep (no `constellation|Stellarium|AGPL` match anywhere
in NOTICE.md):

- **`constellation-lines.json` has no NOTICE entry.** Per its own generator
  (`tools/build-data/constellations.mjs:9–14, 113–115`) the line topology comes from
  **Stellarium's `western` sky culture (AGPL-3.0)** with vertex positions from
  **HYG v4.1 (CC BY-SA 4.0)**. Both require attribution; AGPL is strong copyleft.
  Whether AGPL's terms bind a *data extraction* like this is genuinely murky —
  Stellarium's skyculture index/description files are the part carrying the license —
  but the fix is cheap either way: document it in NOTICE.md, or regenerate the lines
  from a permissively-licensed source (e.g. the `d3-celestial` / Stellarium-derived
  datasets published under CC/MIT variants) if the copyleft is unwanted.
- **The required attribution is surfaced nowhere in the app.** NOTICE.md:157 says the
  HYG CC BY-SA line must appear in "the app's about/credits screen once one exists" —
  no credits/about UI exists in any component (verified). A modal or footer link
  rendering NOTICE.md's text satisfies both HYG and Stellarium attribution in one go.
- **Root `README.md:32` still says `License: TBD`** while the repo bundles
  share-alike (CC BY-SA) and possibly-AGPL data. Pick the project license — and note
  that if the AGPL reading of the constellation data is taken seriously, it
  constrains the choice for the bundle.

## 6. Correctness & robustness

The math core is in excellent shape; these are the residual risks, ranked.

### 6.1 The ΔT override is load-bearing and untested

`astronomyEngineDeltaT.ts:29` globally overrides astronomy-engine's ΔT to the locked
69.1 s via `SetDeltaTFunction`, activated only by bare side-effect imports. Coverage
today is complete — both astronomy-engine importers (`stores/skyView.ts`,
`stores/localCircumstances.ts`) import it — but the mechanism is silent-failure by
design: any *future* file that imports astronomy-engine without the side-effect
import reintroduces a ~6.3 s TT skew that visibly separates the Sun/Moon disks
during totality (the exact bug already fixed once, PLAN.md:1593–1635).
**Add one test** that imports astronomy-engine the same way the stores do and asserts
a Sun position consistent with ΔT=69.1 (or asserts via a probe ΔT call); it turns a
silent regression into a red test.

### 6.2 Known data inconsistencies to close or accept explicitly

- **`eclipse-times.json` was generated at ΔT=69.2 s** (ytliu source) vs. the app's
  69.1 s — a documented ~3 s / 0.06° mismatch (NOTICE.md:46–47). Either regenerate
  from eclipse-calc at 69.1 or add the caveat to the table's UI tooltip. Small, but
  it's the one number on screen that doesn't share the app's time base.
- **`generate_elevation.py:63–64` claims "Same bbox as basemap.mjs" but isn't**:
  elevation covers `-10.5,35 → 6.5,44.5`; the basemap covers `-10.5,35 → 9.5,51.5`
  (verified). Practical exposure is small (the eclipse strip is inside the elevation
  bbox) but the map lets you click anywhere on the wider basemap — **verify what
  `elevationAt` returns outside its grid** (clamp? undefined? NaN into the contact
  solver?) and add a test pinning that behavior. Fix the comment regardless.
- **Residual ~2.5″ Sun–Moon separation** at C2/C3 from astronomy-engine's analytic
  series vs. DE440s (astronomyEngineDeltaT.ts:17–20) — inherent, invisible at UI
  scale. Recommend: record as *accepted* in §14 and cut the Chebyshev upgrade (§2.3).

### 6.3 Minor numerical notes (no action needed, for the record)

- `findMaximumTime`'s ±12 h coarse scan at 0.5 h steps (localCircumstances.ts:188–199)
  assumes a wide minimum basin — fine for this event, golden-tested; would be brittle
  for grazing geometries in a reused-for-2027 scenario.
- `obscurationAt`'s 4 h validity window, the `!(radical >= 0)` NaN-catching
  comparison (shadowOutline.ts:89, 245), and the Newton+bisection safeguards are all
  deliberate and correct — good work worth keeping as-is.
- Stars are fed to `Horizon()` as unprecessed J2000 (skyView.ts:3–6) — documented,
  sub-degree, fine for mag<3 dots.

## 7. Testing — invert the coverage

Current coverage is a mirror image of where bugs have actually occurred:

- **Covered (well):** polynomial evaluation, observer geometry, contact times,
  shadow outlines — golden fixtures from the Python oracle, tight tolerances, plus a
  genuine regression test for the terminator flicker (shadowOutline.test.ts:118–144).
- **Uncovered (entirely):** all 8 Svelte components, all 7 stores, `lib/format.ts`,
  `lib/actions/resizable.ts`.

The empirical bug history says the risk is in the uncovered half: both user-caught
field bugs (countdown counting down to an unobservable C2; sim mode desync) were in
**store/component glue**, not math. Specifically untested today:

1. **Sunset filtering / "next observable event"** — the logic is *triplicated* in
   ContactsPanel.svelte:48–51, CountdownPanel.svelte:30–57, and TimeBar.svelte:147–155,
   and it already produced two real bugs. Extract it into one tested pure function
   (`eclipse/observable.ts`: `(contacts, sunset) → visible events + next event`) and
   the three call sites become trivial (also fixes the §8 duplication).
2. **The ΔT override installation** (§6.1).
3. **`stores/localCircumstances.ts`'s sunset interleaving** (`SearchRiseSet` glue,
   localCircumstances.ts:42–53).
4. `format.ts` countdown/duration edge cases (negative, >24 h, sub-minute).

That's roughly a day of work for the four items. Component-level DOM testing
(Testing Library / Playwright) is **not** recommended at T−30 — the two field
rehearsals (§3) are the higher-value integration test.

Also note: the golden fixtures regenerate only via the undocumented conda env (§9),
but since fixtures are committed, `npm run test` works on a clean clone — keep it
that way (never gitignore `app/test/fixtures/`).

## 8. Architecture & code quality (below-the-line priority)

Real observations, deliberately ranked behind everything above. None of these should
displace §§2–7 work before the event.

1. **The plan says Canvas 2D; the app is 100% SVG** (PLAN.md:109, 251, 274 vs. zero
   `getContext` calls in `app/src`). SVG is working fine at this scale. *Do
   differently:* update §14's stack decision to match reality — don't rewrite the
   renderer. Revisit only if profiling (below) says otherwise.
2. **Sim playback recomputes the world at animation-frame rate.** `TimeBar.clockTick`
   advances `simTimeMs` per rAF (TimeBar.svelte:255–272) → `effectiveTime` →
   `skyView` recomputes `Horizon()` for 174 stars + every constellation vertex +
   5 planets + Sun/Moon **per frame** (skyView.ts:147–176), and MapPanel recomputes
   60+240-point shadow outlines per tick (MapPanel.svelte:236–259). It apparently
   holds 60 fps on the dev machine — but **profile it once on the actual field
   laptop**; if it stutters, the cheap fix is splitting `skyView` into a slow
   (stars/planets, 1 Hz) and fast (Sun/Moon, per-frame) store, not a Canvas rewrite.
   Live mode is 1 Hz and unaffected.
3. **`localCircumstances` re-root-finds on every `pointermove`** during map drag
   (contact search + `SearchRiseSet` per move event). If drag feels janky in the
   rehearsal, debounce to pointer-up or ~100 ms.
4. **Triplicated logic** — sunset filtering (fix via §7.1); tangent-plane projection
   in three places (CountdownPanel.svelte:174–180, SkyPanel.svelte:185–188, 198–206);
   Sun/Moon/sky color literals duplicated across panels instead of `app.css` vars.
5. **Diagnostic/dead code shipping:** MapPanel's ~1,500 per-sample `<circle>` dots
   are explicitly "deliberately diagnostic, not final polish" (MapPanel.svelte:50–53)
   — remove in polish week (also a DOM-size win). SkyPanel's disabled tuning panel
   (`SHOW_TUNING_PANEL = false`, three `$state` margins + markup + CSS,
   SkyPanel.svelte:149–159, 440–458, 587–621) — delete; git remembers. TimeBar's
   hardcoded 15:30–20:00 zoom-out window is self-labeled "will be removed later"
   (TimeBar.svelte:19–22) — derive from contact times or keep and delete the apology.
6. **Component size:** MapPanel 708 lines, SkyPanel 622, TimeBar 605 — the inline
   projection setups (Spain Mercator fit; custom oblique stereographic) are
   module-worthy (`lib/projections.ts`). Worth doing *only* alongside other edits;
   don't refactor for its own sake at T−30.
7. **Stores are all classic `svelte/store` while components are all runes** — a
   consistent, working split; leave it alone until after the event.

## 9. Data pipeline — make it reproducible by strangers (including future you)

Everything shipped in `app/src/data/` is committed, so the *app* builds without this
section being fixed. But the pipeline itself currently requires tribal knowledge,
which fails exactly when a data file needs urgent regeneration in week 3:

1. **No "regenerate everything" entry point.** `tools/build-data/package.json:7–14`
   scripts cover only the 6 JS tools; the 5 Python generators and `tools/gen-vectors`
   are run by hand, in an order nobody wrote down — notably
   **`generate_eclipse_times.py` reads the committed `shadow-frames-global.json` as
   input** (generate_eclipse_times.py:101, 149), so it must run *after*
   `generate_shadow_frames_global.py`. Write `tools/build-data/README.md` + a
   `regen-all` script encoding the order.
2. **The Python env is undocumented.** No `environment.yml`/`requirements.txt`
   anywhere; implicit deps: skyfield, scipy, numpy, netCDF4, plus the **sibling
   `../eclipse-calc` repo** installed editable. Two scripts even disagree on its
   relative path (`generate_besselian.py:9` says `../eclipse-calc`;
   `generate_shadow_frames.py:11` says `../../../eclipse-calc`). Export the conda env
   to a committed `environment.yml` and reconcile the comments.
3. **Hardcoded machine-specific ephemeris path** as default in all five Python
   scripts (`DEFAULT_EPHEMERIS = C:\Users\lauri.kangas\OneDrive\python\eclipse\de440s.bsp`,
   e.g. generate_besselian.py:24) — the `ECLIPSE_CALC_EPHEMERIS` override exists;
   make the default a repo-relative path or fail with a clear message.
4. **Un-scripted manual `curl` steps** for the `.cache/` inputs (HYG CSV 34 MB,
   Natural Earth places/roads, ETOPO chunks, Stellarium index — sources listed only
   in code comments, e.g. cities.mjs:10, roads.mjs:11–12, stars.mjs:9). Fold the
   downloads into the scripts (they're build-time; network is allowed there).
5. The one *output* defect the pipeline produced — the elevation bbox comment — is
   §6.2's item; the NOTICE gap is §4's.

## Appendix A — What is genuinely good (keep doing this)

Credit where due, because the "do differently" list above is only half the picture:

- **Oracle-driven validation.** Golden fixtures generated from an independent,
  separately-tested Python implementation, with sub-millisecond contact agreement,
  is the right architecture for safety-of-observation math, executed properly.
- **Bugs get root-caused, fixed at the source, and pinned with regression tests** —
  the NaN flicker fix (three-layer fix + per-second sweep test) and the ΔT
  unification (with its exemplary explanatory docblock) are model examples.
- **Offline discipline held in the shipped code**: zero runtime fetches, system font
  stacks, all data static-imported into one self-contained 1.1 MB bundle — verified,
  not just intended.
- **Honest placeholder hygiene** — placeholders are labeled in-code ("Not yet
  available", "will be removed later") rather than passed off as real.
- **License bookkeeping** (10 of 11 files) in NOTICE.md is more thorough than most
  commercial projects manage.
